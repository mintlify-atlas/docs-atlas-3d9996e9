## Overview

The Real Madrid RAG application supports two configuration methods:

1. **appsettings.json** - File-based configuration (recommended for development)
2. **Environment Variables** - Runtime configuration (recommended for production)

Environment variables take precedence over appsettings.json values.

## Configuration Structure

The application uses three main configuration sections:

- **AzureOpenAI** - Azure OpenAI service settings
- **AzureAISearch** - Azure AI Search settings
- **SystemPrompt** - RAG system prompt customization

## appsettings.json Configuration

Create or modify `appsettings.json` in the application root directory:

```json
{
  "AzureOpenAI": {
    "Endpoint": "https://<tu-openai>.openai.azure.com/",
    "ApiKey": "<tu-api-key>",
    "ChatDeployment": "chat",
    "EmbeddingDeployment": "embedding",
    "ApiVersion": "2024-10-21",
    "MaxTokens": 800,
    "Temperature": 0.1
  },
  "AzureAISearch": {
    "Endpoint": "https://<tu-search>.search.windows.net",
    "ApiKey": "<tu-api-key>",
    "IndexName": "rm-rag-index",
    "Top": 5
  },
  "SystemPrompt": "Eres un asistente experto de Real Madrid. Responde solo con el contexto recuperado y sé claro y conciso."
}
```

<Warning>
Never commit `appsettings.json` with real API keys to version control. Use environment variables or Azure Key Vault for production deployments.
</Warning>

## Configuration Loading Order

The application loads configuration in the following order (later sources override earlier ones):

1. Default values from [AzureOptions.cs](/home/daytona/workspace/source/src/RagConsole/Config/AzureOptions.cs)
2. Values from `appsettings.json`
3. Environment variables (highest priority)

The loader searches for `appsettings.json` in multiple locations:

- Current working directory
- Application base directory
- Parent directory (for development scenarios)

See [AzureOptionsLoader.cs:43-78](/home/daytona/workspace/source/src/RagConsole/Config/AzureOptionsLoader.cs:43) for implementation details.

## Configuration Validation

The application validates required configuration on startup. Missing or invalid values will cause the application to exit with error messages.

### Required Fields

The following fields are validated as required (see [AzureOptions.cs:9-44](/home/daytona/workspace/source/src/RagConsole/Config/AzureOptions.cs:9)):

- `AzureOpenAI.Endpoint`
- `AzureOpenAI.ApiKey`
- `AzureOpenAI.ChatDeployment`
- `AzureAISearch.Endpoint`
- `AzureAISearch.ApiKey`
- `AzureAISearch.IndexName`

<Note>
Placeholder values containing `<` or `>` characters (like `<tu-openai>`) are treated as missing values.
</Note>

### Validation Errors

If validation fails, the application displays errors like:

```
Configuración incompleta. Revisa appsettings.json o variables de entorno:
- AzureOpenAI.Endpoint es requerido.
- AzureAISearch.ApiKey es requerido.
```

See [Program.cs:8-16](/home/daytona/workspace/source/src/RagConsole/Program.cs:8) for validation implementation.

## Security Best Practices

<Warning>
**Never commit secrets to version control**

Always use secure secret management:
- Azure Key Vault for production
- Environment variables for local development
- `.gitignore` to exclude `appsettings.json`
</Warning>

### Recommended Practices

1. **Use Managed Identities** - For Azure-hosted applications, use Azure Managed Identity instead of API keys
2. **Rotate Keys Regularly** - Periodically regenerate API keys in the Azure Portal
3. **Principle of Least Privilege** - Use read-only keys where possible
4. **Separate Environments** - Use different Azure resources for dev/staging/prod

## Troubleshooting

### Configuration Not Loading

If your configuration isn't loading:

1. Verify `appsettings.json` is in the correct location
2. Check file permissions (must be readable)
3. Validate JSON syntax using a JSON validator
4. Review console output for validation errors

### Environment Variables Not Working

If environment variables aren't being recognized:

1. Verify variable names match exactly (case-sensitive on Linux)
2. Restart your terminal/IDE after setting variables
3. For systemd services, ensure variables are in the service file
4. Check for typos in variable names

### API Key Issues

<Warning>
API keys must not contain `<` or `>` characters. These are treated as placeholder values and will fail validation.
</Warning>

If you encounter API key validation errors:

1. Regenerate keys from Azure Portal
2. Ensure no extra whitespace or quotes around keys
3. Use the primary key (key1) from Azure
